# 云集智能音乐创意台 - 三目录文件夹结构

> 本文档参考云集智能视频创意站的结构设计，明确项目的三个核心目录及其职责。

---

## 三目录总览

```
云集智能音乐创意台/
│
├── dev/            # ⭐ 开发目录 - 源代码（Git管理）
│   └── dist/       # ⭐ 分发目录 - EXE打包输出（Git不管理，位于 dev/ 下）
└── test/           # ⭐ 测试目录 - 测试脚本与测试文件
```

---

## 一、dev/ — 开发目录

### 定位
**唯一的日常开发工作区**，所有源代码的编写、修改和调试都在此目录进行。

### Git管理
✅ **Git管理** — 推送到远程仓库

### 目录结构
```
dev/
├── app/                              # 项目源代码
│   ├── main.py                       # 启动器主程序（PyQt6 GUI）
│   ├── version_manager.py            # 版本管理器
│   ├── build_sfx_exe.py             # EXE构建脚本（onedir + 7z SFX）
│   ├── icon.ico                      # 应用图标
│   ├── ico.png                       # 高清图标
│   ├── splash.png                    # 启动画面
│   ├── requirements.txt              # Python依赖
│   ├── version_history.json          # 版本历史
│   ├── versions.json                 # 版本列表
│   ├── ace-step-ui/                  # 青龙训练器前端
│   ├── acestep/                      # 核心后端
│   └── scripts/                      # 启动脚本（仅脚本，不含重复项目文件）
│       ├── 1、install-uv-qinglong.ps1
│       ├── 2、run_gradio.ps1
│       ├── 3、run_server.ps1
│       ├── 4、run_npmgui.ps1
│       ├── 5、run_qinglong_backend.ps1
│       └── 6、run_qinglong_frontend.ps1
│
├── data/                             # 用户数据（Git不管理）
│   ├── config/                       # 运行时配置
│   ├── outputs/                      # 生成输出
│   ├── uploads/                      # 上传文件
│   ├── models/                       # AI模型文件
│   └── .venv/                        # Python虚拟环境
│
└── temp/                             # 临时文件（Git不管理，可删除）
    ├── logs/                         # 日志文件
    ├── cache/                        # 缓存文件
    └── debug/                        # 调试文件
```

### 职责
| 职责 | 说明 |
|------|------|
| 日常编码 | 所有功能开发、Bug修复在这里进行 |
| 本地测试 | 代码修改后在此目录内测试运行 |
| 版本构建 | 从 `dev/app/` 直接构建EXE，输出到 `dev/dist/` |
| Git提交 | `dev/` 目录提交并推送到远程仓库 |

### 重要原则
- ❌ **不要在 `dev/app/scripts/` 下放置重复的项目文件**（如 main.py、ace-step-ui/、acestep/）
- ✅ `dev/app/scripts/` 只放启动脚本
- ❌ `dev/data/` 和 `dev/temp/` 不纳入Git管理

## 二、dev/dist/ — 分发目录

### 定位
**EXE打包输出目录**，存放构建脚本生成的 `.exe` 文件。

### Git管理
❌ **Git不管理** — 被 `.gitignore` 忽略

### 目录结构
```
dev/dist/
└── 云集智能音乐创意台-vYYYY.MM.DD.HHMM.exe     # 构建输出的EXE
```

### 职责
| 职责 | 说明 |
|------|------|
| EXE存储 | `build_sfx_exe.py` 执行后EXE输出到 `dev/dist/` |
| 分发准备 | 从此目录提取EXE用于发布 |
| 历史存档 | 保留各版本的EXE文件作为存档 |

### 与旧结构的区别
| 旧结构 | 新结构 | 说明 |
|--------|--------|------|
| EXE放在 `dev/` 根目录 | EXE放在 `dev/dist/` 目录 | 分离源码和构建产物 |
| `ver/` 存放稳定版EXE | `ver/` **保持不变** | 手动放入的正式版本，与 dev/dist/ 分开 |
| `pack/` 存放发布包 | `pack/` 保持不变 | 本地发布整合包 |

> ⚠️ **`dev/dist/` 与 `ver/` 的区别**：
> - `dev/dist/` — 构建脚本自动输出，每次 `build_sfx_exe.py` 都往这里写
> - `ver/` — 正式版本分发目录，**由开发者手动放入**经过验证的稳定版EXE

### 构建命令
```bash
cd dev/app/
python build_sfx_exe.py
```
构建完成后，EXE自动输出到 `dev/dist/` 文件夹。

## 三、test/ — 测试目录

### 定位
**独立的测试工作区**，存放所有测试脚本、测试配置和测试数据。

### Git管理
✅ **Git管理**（测试代码，可选纳入Git） — 测试脚本应纳入版本管理，临时测试数据可忽略

### 目录结构
```
test/
├── test_launcher.py                  # 启动器测试
├── test_build.py                     # 构建脚本测试
├── test_crash.py                     # 崩溃恢复测试
├── test_*.py                         # 其他测试脚本
├── test_*.bat                        # 批处理测试脚本
├── fixtures/                         # 测试数据/夹具
│   ├── sample_audio.mp3              # 测试用音频
│   ├── sample_config.json            # 测试用配置
│   └── ...
└── outputs/                          # 测试运行输出（Git不管理）
    └── ...
```

### 职责
| 职责 | 说明 |
|------|------|
| 单元测试 | 核心模块的单元测试脚本 |
| 集成测试 | 多模块联动的集成测试 |
| 测试数据 | 测试用的模拟数据和配置文件 |
| 构建验证 | 打包后EXE的功能验证测试 |

### 重要原则
- ✅ 测试脚本文件以 `test_` 前缀命名
- ✅ 测试数据放在 `test/fixtures/` 子目录
- ✅ 测试输出放在 `test/outputs/`（Git不管理）
- ❌ **不要在 `dev/` 目录中散落测试文件**
- ❌ 不要将测试用的模型文件或大体积依赖放在 test/

## 四、三目录协作流程

### 完整开发→测试→打包流程

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│    dev/      │ ───→ │    test/     │ ───→ │ dev/dist/  │
│  开发编码     │      │  测试验证     │      │  EXE打包输出  │
└──────────────┘      └──────────────┘      └──────────────┘
       │                      │                      │
       │ 1. 在dev/app/中编码   │ 2. 在test/中运行测试  │ 3. 构建EXE输出到dev/dist/
       │   修改源代码          │   验证功能正确性       │   准备分发
```

### 详细步骤

**步骤1: 开发（在 dev/ 目录）**
```
1. 在 dev/app/ 中修改源代码
2. 运行 dev/app/ 中的程序进行调试
```

**步骤2: 测试（在 test/ 目录）**
```
1. 将测试脚本放在 test/ 目录
2. 运行测试验证功能和稳定性
3. 测试通过后进入打包阶段
```

**步骤3: 打包（EXE输出到 dev/dist/ 目录）**
```
1. 运行: cd dev/app/ && python build_sfx_exe.py
2. EXE自动生成到 dev/dist/ 目录
3. 从 dev/dist/ 提取EXE进行发布分发
```

## 五、Git管理总结

| 目录 | Git管理 | 说明 |
|------|---------|------|
| **dev/app/** | ✅ 管理 | 源代码，推送远程 |
| **dev/app/scripts/** | ✅ 管理 | 启动脚本 |
| **dev/data/** | ❌ 不管理 | 用户数据、模型、虚拟环境 |
| **dev/temp/** | ❌ 不管理 | 临时文件 |
| **dev/dist/** | ❌ 不管理 | EXE构建产物 |
| **test/** | ✅ 管理（推荐） | 测试脚本纳入版本管理 |
| **test/outputs/** | ❌ 不管理 | 测试运行输出 |
