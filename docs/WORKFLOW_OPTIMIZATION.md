# 启动器开发工作流程（最终版）

## 优化后的项目结构

```
云集智能音乐创意台/
├── build/                       # 所有版本的开发文件夹（每个版本独立）
│   ├── 云集智能音乐创意台-v2026.03.28.2132/
│   │   ├── main.py
│   │   ├── version_manager.py
│   │   ├── icon.ico
│   │   ├── launcher_config.json
│   │   ├── requirements.txt
│   │   └── launcher.spec
│   ├── 云集智能音乐创意台-v2026.03.28.2223/
│   │   ├── main.py
│   │   ├── version_manager.py
│   │   ├── icon.ico
│   │   ├── launcher_config.json
│   │   ├── requirements.txt
│   │   └── launcher.spec
│   └── ...
├── dist/                        # 所有版本的EXE输出目录（统一）
│   ├── 云集智能音乐创意台-v2026.03.28.2223.exe
│   └── ...
├── build-version.py            # 版本化构建脚本
└── docs/                       # 文档（保留）
```

## 核心原则

1. **每个版本独立文件夹** - `build/` 下每个版本都是完整的开发文件夹
2. **没有"当前开发版本"** - 所有开发都在具体的版本文件夹中进行
3. **默认使用最新版本** - 不指定版本时，自动在最新版本基础上开发
4. **统一EXE输出** - 所有EXE都在根目录 `dist/`（版本文件夹内不保留dist）
5. **简化版本文件夹结构** - 直接就是launcher文件，没有 `src/`、`config/` 嵌套

## 新的开发工作流程

### 场景1：在最新版本基础上开发新版本
```bash
# 在项目根目录运行
python build-version.py
```

这个脚本会：
1. 自动找到 `build/` 目录下最新的版本文件夹
2. 复制最新版本创建新版本文件夹（带当前时间戳）
3. 在新版本文件夹中构建EXE
4. 将EXE移动到根目录 `dist/` 文件夹
5. 清理版本文件夹内的临时构建文件（build和dist）

### 场景2：在特定版本基础上开发
**重要：先创建新版本文件夹，再进行修改！**

1. **复制该版本文件夹创建新版本**
   - 找到 `build/` 下对应的版本文件夹
   - 复制创建新版本文件夹（带当前时间戳）
   - 例如：`build/云集智能音乐创意台-v2026.03.30.1501/` → `build/云集智能音乐创意台-v2026.03.30.1624/`

2. **在新版本文件夹中进行开发**
   - 只修改新版本文件夹中的文件
   - 确保功能正常
   - 测试验证

3. **使用该新版本文件夹构建EXE**
   ```bash
   python build-version.py 云集智能音乐创意台-v2026.03.30.1624
   ```

**⚠️ 禁止直接修改旧版本文件夹！必须先创建新版本再修改！**

### 场景3：回滚到旧版本
1. 直接使用 `build/` 目录下对应的旧版本文件夹
2. 可以在旧版本基础上修改和构建
3. 不需要复杂的Git操作

## 版本文件夹结构

每个版本文件夹都是完整的、可独立运行的：
```
云集智能音乐创意台-vYYYY.MM.DD.HHMM/
├── main.py                 # 主程序
├── version_manager.py       # 版本管理器
├── icon.ico                # 窗口图标
├── launcher_config.json    # 配置文件
├── requirements.txt        # 依赖
└── launcher.spec           # PyInstaller配置
```

## 优势

1. **完全独立** - 每个版本都是完整的开发环境，互不干扰
2. **易于回滚** - 直接使用旧版本文件夹即可
3. **清晰明了** - 所有版本都在 `build/` 目录，一目了然
4. **简化结构** - 版本文件夹内没有嵌套，直接就是launcher文件
5. **统一输出** - 所有EXE都在 `dist/`，版本文件夹内不保留dist
6. **避免重复** - 不会有多个dist文件夹分散在各处

## Git提交建议

提交时应该包含：
- `build/` 目录下的新版本文件夹
- `dist/` 目录中新的EXE文件（如果需要发布）
- 不应该提交临时构建目录（每个版本文件夹内的build和dist会被自动清理）

## 回退方案

原有的构建脚本仍然保留参考：
- `build-new-version.py` - 原有的完整构建脚本
