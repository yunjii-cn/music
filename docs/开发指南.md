## 🎯 新的开发工作流：

```
1. 日常开发 → **直接在 dev/app/ 修改和测试**
   - dev/ 是日常开发目录，唯一的工作区
   - 直接修改 app/ 下的文件
   - 不需要频繁创建版本！

2. 测试 → 在 dev/ 测试

3. 保存版本（只有测试通过后想保存时）
   - 把 dev/app/ 复制到 build/ 创建新版本
   - build/ 只保存版本历史（不含 .venv/.cache 等临时文件）

4. 回滚版本 → 从 build/ 的旧版本复制到 dev/app/

5. 发布稳定版 → 从 build/ 选稳定版复制到 pack/

6. Git提交 → dev/ 目录提交并推送到远程
```

## 开发工作流

### ⭐ 标准开发流程（推荐）

#### 阶段1: 日常开发（在 dev/app/ 修改）
```
✅ 日常开发 → 在 dev/app/ 目录修改和测试
   - dev/app/ 是开发测试目录
   - 可以从 build/ 的某个版本复制过来作为起点
   - 修改完成后在 dev/app/ 测试
```

#### 阶段2: 保存版本（复制到 build/）
```bash
# 测试通过后，在 build/ 目录创建新的时间版本文件夹
# 将 dev/app/ 内容复制到新版本文件夹

# 版本号命名严格规则：
# 格式：云集智能音乐创意台-vYYYY.MM.DD.HHMM
# YYYY.MM.DD：年.月.日
# HHMM：创建文件夹时的具体时间（24小时制，小时分钟）
# ✅ 正确：云集智能音乐创意台-v2026.03.30.2206
# ❌ 错误：云集智能音乐创意台-v2026.03.31.0000 (手动编号)
```

#### 阶段3: 回滚版本（从 build/ 复制）
```bash
# 如果需要回滚，从 build/ 的旧版本复制到 dev/app/
# build/ 保存了所有历史版本，可以随时回滚
```

#### 阶段4: 发布稳定版（复制到 pack/）
```bash
# 测试完全通过后，从 build/ 创建纯净整合包到 pack/
# pack/ 是本地纯净整合包，提供给用户
```

#### 阶段5: Git提交（提交 dev/）
```bash
# 将 dev/ 目录提交并推送到远程
# dev/ 包含EXE和app/，推送到远程仓库

# ⭐ 重要：根据实际工作内容写提交信息，不要只写"更新"
# 正确示例：
git add dev/
git commit -m "v2026.04.14.0600: 互换软件更新和模型管理按钮位置"
git commit -m "v2026.04.14.0600: 添加青龙训练器模型检测功能"
git commit -m "v2026.04.14.0600: 完善青龙训练器中文翻译"

# 自动推送（可选，但推荐手动确认后再推送
git push
```

## 分支管理策略

### 分支说明

| 分支 | 用途 | 操作方式 |
|------|------|----------|
| **main** | 开发分支 | 自动打包上传 |
| **beta** | 测试分支 | 手动合并和推送 |
| **stable** | 稳定版分支 | 手动合并和打标签 |

### 分支工作流

```
main (开发) 
  ↓ (测试通过后)
beta (测试)
  ↓ (充分测试后)
stable (稳定版 + 标签)
```

### 详细操作

#### 1. 开发分支 (main)
- **日常开发在此分支进行**
- 运行 `python build-version.py` 会自动提交和推送
- 适用于快速迭代和功能开发

#### 2. 测试分支 (beta)
```bash
# 切换到beta分支
git checkout beta

# 合并main分支的更改
git merge main

# 手动测试
# ... 测试 ...

# 手动推送
git push origin beta
```

#### 3. 稳定版分支 (stable)
```bash
# 切换到stable分支
git checkout stable

# 合并beta分支的更改
git merge beta

# 打版本标签
git tag -a v1.0.0 -m "Release v1.0.0"

# 推送标签
git push origin stable --tags
```

### 📦 发布打包工作流程

```

```

## AI助手上下文指南

### 首次接触项目

当新的AI助手开始工作时，请按以下步骤操作:

1. **阅读本文档** - 了解项目结构
2. **查看 `build/` 目录下最新版本** - 理解完整项目结构
3. **查看 `dev/` 目录** - 理解纯净整合包结构
4. **不要修改配置文件** - 除非明确要求

### 修改原则

- ✅ **日常开发在 dev/app/ 目录进行**
- ✅ **测试通过后复制到 build/ 保存版本**
- ✅ **需要回滚时从 build/ 的旧版本复制到 dev/app/**
- ✅ **发布稳定版时从 build/ 创建纯净整合包到 pack/**
- ✅ **Git只提交 dev/ 的完整内容**
- ✅ **所有开发版本保存在 build/ 目录的版本文件夹中**

### 避免的操作

- ❌ 不要在根目录创建项目文件
- ❌ 不要修改 build/ 中的旧版本（除非回滚）
- ❌ 不要直接修改 pack/ 中的文件
- ❌ 不要删除 build/ 中的历史版本
- ❌ 不要跳过本文档直接开始编码
- ❌ 不要在版本文件夹内保留临时的 build/ 和 dist/
