## 可行性评估：改造训练链路支持 XL (acestep-v15-xl-turbo) 底座
- 用户问"官方都还没支持？能改造成支持XL训练吗？" → 已深挖训练代码给出判断。
- **结论：能改，且主要是低风险的"变体注册"改造，不是重写训练数学。** 关键证据：
  - 加载器 `training_v2/model_loader.py:load_decoder_for_training` 用 `AutoModel.from_pretrained(trust_remote_code=True)` + HF `auto_map` → 靠 config 自动实例化架构类，**没写死 `AceStepConditionGenerationModel`**。XL 的 `modeling_acestep_v15_xl.py`(4B)作为权重同包文件(model_downloader.py:410 已列)下载后由 auto_map 自动加载。
  - LoRA 注入 `training/lora_utils.py:get_dit_target_modules` 按 **`q/k/v/o_proj` 子串**动态扫 `model.decoder.named_modules()` 找 Linear → **架构无感**（1.5B/4B 都适用），无硬编码层数。
  - 训练器 `trainer_fixed.py` / `FixedLoRAModule` 只操作 `.model`/`.model.decoder`/`.forward(hidden_states, encoder_hidden_states, ...)` → 变体无感。
- **需要改的注册点(全为加法,低风险)**：
  - `model_loader.py` `_VARIANT_DIR` 加 `"xl": "acestep-v15-xl-turbo"`(及 xl-sft/xl-base)。
  - `quick_presets.py` `ALL_VARIANTS`+`VARIANT_DEFAULTS`(xl-turbo: shift 3.0/steps 8; xl-sft/base: 1.0/50)+`_VARIANT_PREFERENCE`。
  - `model_discovery.py` `detect_base_model` 识别 xl（注意 XL-turbo 的 config 含 `is_turbo=True` 当前会被误判为 "turbo"，需补 xl 分支）。
  - `configs.py`/CLI/UI 确保 `model_variant="xl"` 透传到 `load_decoder_for_training`。
  - 前端 `ace-step-ui/data/qualityPresets.ts` 与 `ace-step-ui/server/src/data/qualityPresets.ts`(两份镜像) 加 xl；底座选择 UI 加 xl 选项。
  - 触发 XL 权重下载(~18.8GB bf16,需≥12GB 显存,model_downloader.py:408 已定义)。
- **真正的风险(即"官方没支持"所指,无法静态保证)**：
  - XL 模型代码 `modeling_acestep_v15_xl.py` **不在本仓库**，训练时从 HF `ACE-Step/acestep-v15-xl-turbo` 拉取 → 无法静态验证其 `forward` 签名与 `FixedLoRAModule.training_step` 调用(`hidden_states/encoder_hidden_states/context_latents/attention_mask/encoder_attention_mask`)及 `.decoder`+`q/k/v/o_proj` 暴露方式完全一致。若上游 XL 接口与 1.5B 有差异 → 运行时崩溃或静默产出坏适配器。上游未验证 XL+LoRA。
  - **显存硬门槛**：XL 4B bf16 ≈ 18.8GB 权重，需 fp8(~9.4GB)+gradient_checkpointing+encoder offload 才勉强塞进 ≥12GB；用户本机 VRAM 未知。

## 诊断：青龙后端(api_server)启动即退出（23:07 重启后）
