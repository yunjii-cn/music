## 退出后又自动登录（修复）
- 现象：点"退出"后界面闪一下又回到已登录态。
- 根因（设计缺陷，非偶发）：`AuthContext.logout` 仅清前端 `localStorage` 的 token/user 与 state，`isAuthenticated` 变 false → `LoginGate` 的 `useEffect`（依赖 isAuthenticated）立即重跑 → 未登录分支整页跳 `music.yunjii.cn/login`。而 **UM/微信的登录 session 在 UM 服务端（cookie）仍活着**，官网发现微信在线便直接带着 `yunji_user` 自动回跳 → `LoginGate` 检测 URL 的 yunji_user → `loginWithUM` → 又登录。本质：退出只清了我方本地态，没让 UM 侧登出；且退出后无条件跳 UM 等于"退出即重新发起登录"。
- 修复（不让退出自动重登）：
  1. `context/AuthContext.tsx` 的 `logout`：退出时 `sessionStorage.setItem('yunji_manual_logout','1')` 打"主动退出"标记。
  2. `LoginGate.tsx`：① 顶部加 `MANUAL_LOGOUT_KEY` 常量；② effect 在 `isValidating/isAuthenticated` 检查后、`new URLSearchParams` 之前插入：读 `yunji_manual_logout` 标记，若存在则 `setStatus('loggedout')` 并 **return，不跳 UM**（避免 UM 自动回跳造成死循环）；③ `goLogin` 开头 `removeItem('yunji_manual_logout')`，使点"重新登录"时回跳的 yunji_user 能被正常处理；④ 渲染新增 `loggedout` 状态：紫色登出图标 + 文案"你已退出登录" + "重新登录"按钮（调 goLogin），与 error 状态共用该按钮。
  3. `statusText` 增加 `loggedout` 分支文案。
- 效果：退出后停在"已退出"页，不再自动跳 UM、不再被 UM 微信态秒登；用户主动点"重新登录"才跳 UM（此时 UM 因微信在线仍会秒登，属预期）。
- 遗留/可选增强：若要"彻底登出微信"，需 UM 侧提供 logout 端点（如在 `https://music.yunjii.cn/logout` 注销 cookie 后 redirect 回来），可在 `goLogin` 或 `logout` 时一并跳转——取决于 UM 是否支持，待用户确认。
- 校验：`npm run build` 通过（exit 0），3070 热更、dist 已重建。
- 改动文件：`context/AuthContext.tsx`、`LoginGate.tsx`。

## 接入 UM 单点登出 um/logout.php（彻底登出，补上上一条遗留点）
- 背景：UM 侧已上线与登录页对称的登出收口 `https://music.yunjii.cn/um/logout.php`，支持 redirect/embed/gate/app 参数。行为：同域→清 session+清 um_sso_token cookie+走 UM end_session 单点登出再回该页；跨域 localhost（启动器）→清 session+显式 setcookie 清 um_sso_token cookie，然后直接回跳启动器地址（不走 UM 回合，因 post_logout_redirect_uri 不认 localhost）；无参→回 HTTP_REFERER 再无则登录页。关键：跨域场景显式清了 um_sso_token，同族站点视为已登出。

  3. `.env.example`：新增 `VITE_UM_LOGOUT_URL` 说明+默认值。

## EXE 体积 37→61MB 排查结论
- 现象：用户问为何以前 37MB、现在 60MB。
- 实测事实：
  - dev/dist 旧包实为 38–39MB（如 v2026.05.26.0447.exe），均为 onefile；新包 61MB。
  - 当前 venv PyQt6 **6.11.0 / Qt 6.11.0**（Qt6 bin 目录 144MB，含 avcodec-61.dll 14MB、opengl32sw.dll 20MB、Qt6Core 10.5MB 等巨型 DLL）。PyInstaller 自动收集 GUI 依赖到的全部 Qt 二进制。
  - **新旧打包的 hidden-import / exclude 列表完全一致**（旧 release.spec 与新建 build_onefile_exe.py 都引 PyQt6/QtCore/Gui/Widgets/sip+loguru+psutil+huggingface_hub 系），故收集逻辑相同 → 体积差只能来自 DLL 本身变大。
  - 旧 38MB 包早于 Qt6 bin 的 5/26 安装时间，是用**更早的 PyQt6（6.11 之前）**打的，Qt 运行时更精简、无 avcodec/opengl32sw。
  - 旧 release.spec 有 `upx=True`，但**本机 venv 现在没有 UPX**（PATH/PyInstaller 目录均无 upx.exe），旧压缩效果现在复现不了。
- 结论：体积差主因 = **PyQt6/Qt6 版本升级导致 Qt 运行时膨胀**；非脚本 import 差异。
- 可选回压手段（用户未拍板）：A 装 UPX 重压；B 降级 PyQt6 到出 37MB 的旧版本；C `--exclude-module` 剔掉用不到的 Qt 子模块（QtMultimedia/Quick/OpenGL）省下 avcodec+opengl32sw。建议优先 A。

## 启动流程优化：首次运行就地运行，消除「二次解包 + 三次启动屏」
- 用户观察（痛点）：双击打包 exe 会先弹**静态启动屏**（PyInstaller 自带 `pyi_splash`，只显示 splash.png、无进度），关 → 再弹一次静态屏，关 → 等很久 → 第三次才出**真实进度条**正常加载。根因：原 `_self_relocate` 首次运行会**部署后重启第二个 exe（固定名入口）并 `os._exit`**，导致一个 onefile exe **被解包运行两次**（每次解 ~50MB + 加载 PyQt6），于是出现「2 个静态 pyi_splash（每个 onefile 进程启动时必闪）+ 1 个真实 QSplashScreen」。两次解包还让"等很久"翻倍。
- 优化（`launcher.py` 的 `_self_relocate`）：
  - 重写 `_ensure_entry` → `_deploy(install_root, run_in_place=False)`。`run_in_place=True` 时：**部署完安装目录后不重启第二个 exe**，由同一进程继续加载，并用 `os.environ["YUNJI_INSTALL_ROOT"]=install_root` 把安装根目录告知后续 `main.get_install_root()`。首次运行走此分支，从「2 进程 / 3 次启动屏」降为「1 进程 / 1 次静态屏(pyi) + 1 次真实进度条(QSplash)」，且 pyi 关闭瞬间 QSplash 已建好=丝滑无缝（即用户要的"等真实进度条打开后再关"）。
  - 其它分支不变：`run_in_place=False`（版本化 exe 在 ver/ 委托启动、或已部署目录存在但入口缺失自修复）仍重启固定名入口+`--cleanup` 删除原始 exe。
  - `get_install_root()`（main.py）新增：frozen 时先读 `YUNJI_INSTALL_ROOT` 环境变量（就地运行设置），命中且是目录则直接返回，避免 exe 仍在便携位置（如 Downloads）时把 app 建到同级目录。
- 文件名保护放宽（launcher 底部）：原来强制 `basename==云集智能音乐创意台.exe` 才放行；现改为放行**合法版本化 exe（正则 `^云集智能音乐创意台-v\d+\.\d+\.\d+`）或固定名入口**，仅当被改成其它名字才弹窗 `os._exit(1)` 拒绝——首次就地运行就是版本化 exe，需放行；版本化 exe 在 ver/ 内仍按规则委托上级固定入口（进程名最终是固定名入口，保护不被篡改）。
- 校验：① 语法 OK；② 隔离单测（抽取 `_self_relocate` 函数本体、桩 `os._exit`/`subprocess.Popen`/文件系统）：首次运行→部署安装根+ver副本+固定入口+version.txt+lock、**设 env、未调 os._exit(未重启)、未 spawn 进程**；`get_install_root()` 读 env 正确指向安装根；合法版本化 exe 放行、篡改名拒绝。③ 规则 A（版本化 exe 在 ver/ 内）→ 委托重启固定名入口（目标确为 `云集智能音乐创意台.exe`）；规则 B（固定名入口本体）→ 直接运行不重启。全部符合预期。
- 说明：首次就地运行**不会删除**用户下载目录里的原始版本化 exe 副本（旧逻辑靠"重启新进程+`--cleanup`"删，现因同进程运行且 `own==cleanup_target` 被跳过保护）。该副本无害，后续用 `安装根/云集智能音乐创意台.exe` 启动即可；用户可手动删下载目录的副本。
